# 輪講まとめ：Uncovering Underexplored Runtime Behaviors in ROS2-Based Autonomous Systems

**論文情報**

- **著者**: Chenghao Fan, Lanshun Nie, Jiacheng Zhang, Kun Dai, Shenghan Gao, Jing Li
- **所属**: Harbin Institute of Technology / New Jersey Institute of Technology
- **掲載**: ACM Transactions on Internet of Things, 2025年10月
- **DOI**: https://doi.org/10.1145/3772083

---

## 1. 研究の背景と動機

ROS2（Robot Operating System 2）は、自律走行車やモバイルロボティクスなど、安全性・性能が重要なドメインで広く使われるミドルウェアである。DDSの上に構築され、非同期通信（Publisher/Subscriber、Service、Action）やExecutorによるコールバックベースの実行を提供する。

**問題意識**: 既存のROS2向けタイミング解析やスケジューリング戦略は、実世界のアプリケーションでは成り立たない**単純化された仮定**に依存している。本研究は、理論モデルと実際のランタイム挙動の間のギャップを、実証的アプローチで明らかにする。

## 2. ROS2の基礎知識（Background）

### 2.1 基本エンティティとExecutorスケジューリング

- **コールバック**: ROS2における最小の実行単位。タイマー・サブスクリプション・サービス・アクションなどで起動される。
- **ノード**: 複数のコールバックを持つ。
- **Executor**: ノードのコールバックを管理・スケジューリング。シングルスレッド型とマルチスレッド型がある。
- **スケジューリング**: ノンプリエンプティブ（非割り込み）。ReadySet内の優先度は `タイマー > サブスクリプション > サービス > クライアント > アクション` の順。

### 2.2 ROS2 Actionエンティティ

- 長時間動作するタスク（例: 経路計画）に使用される通信パターン。
- Action Client と Action Server で構成され、目標送信・フィードバック・結果取得・キャンセルをサポート。
- 内部的にはPublisher/Subscriber/Client/Serviceを複合的に使用。

### 2.3 因果連鎖（Cause-Effect Chain）

- コールバック間のデータ・制御依存の連鎖で、センサ入力からアクチュエータ出力へのデータ伝播を表す。
- **ノード間通信**: DDS経由（Publisher-Subscriber、Client-Service、Action）
- **ノード内通信**: DDS不要の内部バッファ経由（例: TFバッファ）
- **主要メトリクス**:
  - **応答時間（Response Time）**: 連鎖の最初のコールバック開始から最後のコールバック完了まで
  - **最大反応時間（Maximum Reaction Time, MRT）**: 外部イベント発生からアクチュエータに影響が出るまでの最大遅延
  - **最大データ年齢（Maximum Data Age, MDA）**: センサがイベントを検出してから、その影響がアクチュエータで消えるまでの最大期間

### 2.4 TFライブラリ・Navigation2・ビヘイビアツリー

- **TF（Transform Library）**: 座標フレーム間の空間変換を管理。分散バッファ設計で各ノードがローカルバッファを保持。
- **Navigation2**: ROS2ベースのナビゲーションスタック。グローバルプランニング・ローカルプランニング・リカバリ等を提供。
- **ビヘイビアツリー（BT）**: 階層的な行動制御構造。Navigation2のタスク実行のデフォルトメカニズム。

## 3. 主要な発見（4つの実証的観察）

### 3.1 部分的因果連鎖の重要性

**既存の仮定**: タイミング解析はセンサからアクチュエータまでの**エンドツーエンド**連鎖に焦点。

**観察**: 実世界では、エンドツーエンドでない**部分的な因果連鎖**も同様に重要であり、厳しいリアルタイム制約を持つ。

**具体例（Navigation2）**:
- `goal_pose`（目標位置受信）→ Action Server `compute_path_to_pose`（経路計算）→ `goal_response`（応答受信）の部分連鎖に**20msのハードデッドライン**が存在。
- デッドラインを超過すると、**Recoveryメカニズム**が発動（コストマップクリア、スピン、バックアップ、待機など）。
- Recovery発動は通常動作時に深刻な機能劣化（速度低下、振動的挙動）を引き起こす。

**示唆**: リアルタイム解析の対象をエンドツーエンド連鎖だけでなく、部分連鎖にも拡大する必要がある。

### 3.2 因果連鎖の構造・タイミングの実行時変動

**既存の仮定**: 因果連鎖は固定的な構造・ほぼ一定の実行時間を持つ。

**観察**: 実行時に、システムの状態に応じて連鎖の構造とタイミングが**動的に変化**する。

**構造の変動例（Navigation2）**:
- グローバルプランニング起動時: LiDAR → nav2_amcl → nav2_planner → nav2_bt_navigator → nav2_controller → wheels（長い連鎖）
- 巡航状態時: LiDAR → nav2_amcl → nav2_controller → wheels（短い連鎖、Global Plannerをバイパス）
- ビヘイビアツリーのtickにより、1秒ごとにモード遷移が発生。

**タイミングの変動例**:
- Dijkstraベースのグローバルプランニングの実行時間は、目標から遠い時（約180ms）から近い時（約20ms）まで大きく変動。
- WCETでモデル化すると、大部分の実行時間帯で過度に悲観的な解析結果になる。

**示唆**: 動的なモデルと適応的スケジューリング戦略が必要。

### 3.3 非FIFO データアクセスパターン

**既存の仮定**: データバッファはFIFO順序で消費される。

**観察**: 実アプリケーションでは、機能的な整合性のため**意図的にFIFO外のデータアクセス**が行われる。

**具体例（Navigation2のTFバッファ）**:
- ローカルプランニングの`follow_path`コールバックは、最新のLiDARデータと、オドメトリのタイムスタンプに対応する**古いLiDARデータ**の両方を使用。
- `map`フレーム（LiDAR由来）と`odom`フレーム（オドメトリ由来）の時間的整合性を保つため。
- この結果、**MRTとMDAが等しくならない**（FIFO前提では等しくなるが）。

**ROS2の有界キューによる影響**:
- デフォルトのメッセージキューサイズ（10）では、高頻度のPublisherに対して処理が追いつかない場合、古いデータを処理したりデータが欠落する。
- 実験（TurtleBot3 + YOLOv3 Tiny）: カメラ60Hz、バッファサイズ10で**約90%のセンサデータが喪失**。バッファサイズ1では最新データが常に使用される。

**示唆**: タイミング解析フレームワークを非FIFOデータアクセスに対応させる必要がある。

### 3.4 Executorが制御しない計算スレッドの存在

**既存の仮定**: 全ての計算はROS2 Executorの管理下にある。

**観察**: 実アプリケーションでは、多くの重要な計算がExecutor外のスレッドで実行される。

**具体例**:
- **Navigation2のAction Server**: グローバルプランニングの長時間計算を`std::thread`で別スレッドに委任。Executorの非割り込みスケジューリングで他のコールバックをブロックしないため。
- **Costmap2DROSのmapUpdateLoop**: ノード初期化時に起動されるバックグラウンドスレッド。
- **AutowareのNDT Scan Matcher**: OpenMPによる並列スレッド実行。

**問題点**: これらの非管理スレッドはCPUリソースを競合し、ROS2管理下のコールバックとデータをやり取りするが、既存のスケジューリング解析では考慮されていない。

**示唆**: 非Executor制御スレッドをリアルタイム解析にファーストクラスのエンティティとして組み込む必要がある。

## 4. トレーシングツールの拡張（Section 5）

既存の`ros2_tracing`（C. Bédard et al.）を拡張し、以下をサポート:

### 4.1 TFトレーシング

3つの新しいトレースポイントを導入:
- **`store_in_buffer`**: TFバッファへの変換メッセージ格納を記録
- **`calculate_in_buffer`**: 格納されたメッセージの使用を記録
- **`interpolate_in_buffer`**: 2つのTFメッセージの補間を記録

→ ノード内通信（TFバッファ経由）の因果連鎖を再構築可能に。

### 4.2 Actionトレーシング

- Client-Service通信の双方向追跡をサポート。
- 長時間計算スレッド用に`action_callback_start`/`action_callback_end`トレースポイントを追加。

### 4.3 トレーシングオーバーヘッド

追加トレースポイントのオーバーヘッドは平均**0.0012ms**（既存の0.003msより小さい）。最大48バイトのデータを書き込む。

## 5. 実験（シミュレーション環境での検証）

### 5.1 実験環境

- **シミュレータ**: Gazebo 11.10.2 + TurtleBot3 Waffle
- **2台のサーバ**: シミュレーション用とROS2アプリケーション用を分離（1000Mb/s Ethernet直結）
- **ROS2 Humble**, PREEMPT_RTカーネル, CPU周波数固定1.4GHz, コアアイソレーション

### 5.2 スケジューリング構成

| 構成 | 特徴 |
|------|------|
| **Redundant** | 各コンポーネントに十分なリソース。ベースライン。 |
| **Config 1** | nav2_amclとcv_detectionを同一Executor/コアに配置。リソース競合。 |
| **Config 2** | nav2_plannerのAction Serverとcamera_img_recorderを同一コアに配置。プリエンプション導入。 |
| **Config 3** | グローバルプランニングの非管理スレッドとweb_video_serverを同一コアに配置。 |

### 5.3 シナリオ1: 単一目標ナビゲーション + ビデオストリーミング

**主要結果**:

- **Config 1**: Chain 1のMRTが大幅増加（非FIFOデータアクセスにより、最新LiDARデータの遅延が影響）。Chain 2は影響なし（古いデータのみ使用するため）。
- **Config 1**: Chain 1の**59/100インスタンスで1.2sデッドラインを超過**。ミッション完了時間が39s→48.5sに悪化。
- **Config 3**: グローバルプランニングスレッド（Executor外）がweb_video_serverを阻害。ビデオFPSが低下。ロボットが目標に近づくにつれ改善（実行時間の動的変動を反映）。

### 5.4 シナリオ2: 複数チェックポイントナビゲーション

**主要結果**:

- **Config 2**: チェックポイント到着時にcamera_img_recorderとhandle_accept（Action）が同時にReadySetに入り、Executorの優先度でSubscriber > Actionとなるため、handle_acceptが遅延。20msデッドライン超過が約5/100回発生。→ Recoveryメカニズム発動。
- 因果連鎖の構造変動（巡航→グローバルプランニング起動）が影響。

## 6. Autowareケーススタディ（Section 7）

### 6.1 環境

- **シミュレータ**: CARLA 0.9.15 (Town01)
- **自律走行スタック**: Autoware.universe 2023.05
- 車線追従タスクで3つの部分的因果連鎖（LiDAR→制御、IMU→制御、Ego Status→制御）を評価。

### 6.2 結果

| Chain | Config 1 MRT(max) | Config 2 MRT(max) | Best-effort MRT(max) |
|-------|-------------------|-------------------|---------------------|
| Chain 1 (LiDAR) | 1691ms | 567ms | 370ms |
| Chain 2 (IMU) | 2330ms | 1995ms | 3832ms |
| Chain 3 (Ego) | 1718ms | 1483ms | 1873ms |

- **Config 1**: pointcloud_containerとの競合により、Chain 1のMRTが大幅に増加。制御精度が劣化し、車線逸脱が発生。
- **Config 2**: 優先度調整により改善。Chain 2はBest-effortよりも良好（CFSによる不確定な競合が解消）。
- 部分連鎖のスケジューリングが自律走行の機能的正確性に直接影響することを実証。

## 7. 結論と今後の方向性

### 主要貢献

1. **4つの実証的観察**: 部分連鎖の重要性、構造・タイミングの変動、非FIFOアクセス、非管理スレッドの影響
2. **トレーシングツール拡張**: TFとActionをサポートし、現実的な因果連鎖の再構築を実現
3. **実験的検証**: シミュレーションロボットおよびAutowareでの実証

### 今後の研究方向

- 部分連鎖を含む動的モデルの形式化
- 非FIFOデータアクセスに対応したタイミング解析フレームワーク
- 非Executor制御スレッドをスケジューリングモデルに統合
- オンラインモニタリングと適応的スケジューリングの実現
- 有界キューの最適サイズの解析的決定

## 8. 議論ポイント（輪講用）

1. **部分連鎖のデッドライン設定**: 20msのハードデッドラインはNavigation2のデフォルト値だが、アプリケーション固有の要件に基づく適切なデッドライン設定はどうあるべきか？
2. **静的解析 vs 動的解析のトレードオフ**: 実行時変動を考慮すると、静的な最悪ケース解析はどこまで有用か？オンライン解析のオーバーヘッドは許容可能か？
3. **非FIFO問題の一般性**: TFバッファ以外にも、非FIFOアクセスが問題になるケースはどの程度存在するか？
4. **Executorの設計限界**: 非管理スレッドの問題は、ROS2のアーキテクチャ設計の根本的な限界を示唆しているか？将来のROS2バージョンで解決されうるか？
5. **実験の妥当性**: シミュレーション環境での結果が、実機環境にどの程度一般化できるか？
