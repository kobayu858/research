# 輪講資料（簡潔版）：A First Look at ROS 2 Applications Written in Asynchronous Rust

**論文情報**
- 著者：Martin Škoudlil, Michal Sojka, Zdeněk Hanzálek（チェコ工科大学）
- 会議：ECRTS 2025（37th Euromicro Conference on Real-Time Systems）

---

## 1. 論文の概要

### 背景と動機

ROS 2のリアルタイムスケジューリングや応答時間分析（RTA）はこれまでC++を前提に研究されてきた。一方、Rust用ROS 2バインディング「R2R」は非同期（async/await）モデルを採用しており、C++版と実行モデルが大きく異なる。この非同期モデルでリアルタイム性を確保できるかが未解明だった。

### 貢献

1. R2Rおよび複数の非同期Rustランタイムの実行モデルを分析し、C++版と比較
2. 決定論的リアルタイム動作に適したR2Rアプリケーション構造（**futures-rt**）を提案
3. 合成ベンチマークと自動運転ケーススタディで有効性を実証

---

## 2. 前提知識

### 2.1 ROS 2の要点

ROS 2はノードベースのパブリッシュ・サブスクライブ通信フレームワーク。イベント検出を**サンプリング**、コールバック実行主体を**エグゼキュータ**と呼ぶ。

C++（rclcpp）のエグゼキュータは、シングルスレッド（コールバックを型の優先度順に逐次実行）、マルチスレッド（複数スレッドで並列実行、飢餓問題あり）、イベント（DDSから直接イベント受信）の3種類がある。なお、ROS 2 Jazzyで内部データ構造が変更され、同型コールバックの実行順序が予測不可能になった（[#2532](https://github.com/ros2/rclcpp/issues/2532)）。

### 2.2 Rustの非同期プログラミング

Rustの非同期プログラミングでは、`async`/`await`と`Future`トレイトを言語が提供し、実際のスケジューリングは外部ランタイムが担う。非同期タスクはOSスレッドより軽量で、MPSCチャネル（複数送信者→単一受信者のキュー）を介して通信する。

### 2.3 非同期ランタイム比較：futures vs Tokio

**futuresランタイム** — 実装が単純で決定論的リアルタイム向き。ローカルエグゼキュータ（シングルスレッド）は、受信タスクベクタ→アクティブタスクリスト→レディキュー（FIFO）の3構造で動作する。レディキューがFIFO順のため実行順序が予測可能であり、初期化時にメモリ割り当てが集中するためループ内のアロケーションを回避できる。スレッドプールエグゼキュータは複数ワーカーで並列実行するが、実行中に再readyになったタスクを即座に続行するため飢餓問題が発生しうる。`futures::join!()`によるグループ化はグループ内のどのfutureが先にreadyになるかで実行順序が変化し、リアルタイムには不向き。

**Tokioランタイム** — スループット最大化を目的とした設計。LIFOスロット→ローカルレディキュー→グローバルキュー→ワークスティーリングの多段デキューを行う。LIFOスロットによる優先度逆転的挙動、ランダム開始のワークスティーリングなど非決定性が多く、リアルタイムには不適切。

### 2.4 rclcpp・R2R・rclrs 機能比較

| | rclcpp | R2R | rclrs |
|---|---|---|---|
| 言語 | C++ | Rust | Rust |
| 提供元 | ROS公式 | コミュニティ | コミュニティ |
| API設計 | — | 非同期（async/await） | rclcpp類似 |

rclcppは全機能をサポートする成熟した実装。R2Rは通信機能がrclcppとほぼ同等で、非同期スタイルとderive macroによるパラメータ自動生成が強み。rclrsはタイマー未実装・マルチスレッド非対応など制約が大きい。

### 2.5 R2Rの実行モデル

C++との最大の違いは**コールバック実行の管理主体**。C++ではサンプリングもコールバック実行もエグゼキュータが一体的に行うが、R2Rではサンプリングを`Node::spin_once`が、コールバック実行をRust非同期ランタイムがそれぞれ独立に担当する。両者はMPSCチャネルで接続される。

`spin_once`はウェイトセットでイベントを待機し、readyエンティティのイベントをチャネルにプッシュ。チャネル容量は11に固定されており、満杯時はイベントがドロップされる。基本ループは「`run_until_stalled`（初期化）→ ループ内で`spin_once`（サンプリング）→`run_until_stalled`（コールバック実行）」の構造をとる。

R2Rの柔軟な構成により、C++の各種エグゼキュータ構成を模倣できる（シングルスレッド、マルチスレッド相互排他、各コールバック独立など）。ただしサブスクリプションとタイマーの優先度がC++と逆転している点に注意。

## 3. 提案手法：futures-rt構造

### 設計ルール

> **ルール1**：メインスレッドは**最高優先度**で実行し、イベントサンプリング（`spin_once`ループ）に専念する。
>
> **ルール2**：各コールバックは**低優先度の個別スレッド**で、それぞれfuturesローカルエグゼキュータにより実行する。

すべてのスレッドはLinuxの`SCHED_FIFO`スケジューラで動作させる。

### 設計の狙い

メインスレッドを最高優先度にすることで、ROS初期化時に生成されるRMW/DDSスレッドが同じ優先度を継承し、イベント発生後ほぼ即座にサンプリングが行われる。コールバックを低優先度の個別スレッドにすることで、ROS固有の事項から独立した**古典的な単一プロセッサRTAがそのまま適用可能**になる（レートモノトニック優先度割り当て）。

各コールバックスレッドは`spawn_in_thread`で生成され、SCHED_FIFOスケジューラ・指定優先度・専用ローカルエグゼキュータを持つ。スケジューリングは完全にOSの`SCHED_FIFO`に委ねられる。

### 現在の制約

R2Rのチャネル容量が11イベント固定のため、低優先度コールバックでチャネル溢れのリスクがある。

## 4. 実験評価

### 4.1 合成ベンチマーク

5つのトピック（周期10〜200ms、実行時間2〜50ms、CPU利用率90%）を持つパブリッシャー・サブスクライバー構成で、9つのバリアントを比較。AMD Ryzen 7 3700U上の単一隔離CPUコアで20秒間×10回実行。

| バリアント名 | ランタイム | 構成の特徴 |
|---|---|---|
| **futures-rt** | futures | **提案手法**。各コールバックを個別スレッド、RM優先度 |
| futures | futures | ローカルエグゼキュータ、spin_onceと同一スレッド |
| futures-join | futures | スレッドプール、全コールバックをjoin |
| futures-thread-pool | futures | スレッドプール（2スレッド） |
| futures-2-threads | futures | サンプリング専用スレッド + 全コールバック1スレッド |
| **rclcpp-rt** | C++ | 各コールバックに専用エグゼキュータ、RM優先度 |
| rclcpp-st | C++ | デフォルトのシングルスレッドエグゼキュータ |
| tokio / tokio-rt | Tokio | 全スレッド同一優先度 / spinスレッド高優先度 |

**主要な結果（99パーセンタイル エンドツーエンドレイテンシ）：**

1. **futures-rtとrclcpp-rtだけが全トピックのデッドラインを満たした**（デッドライン＝周期）
2. 両バリアントの結果は理論的RTA値と一致
3. トピック5（最低優先度）のみRTA予測よりやや高いレイテンシ（サンプリング・RMW/DDSオーバーヘッドが原因）
4. 他の全バリアントはデッドライン違反。nort-tokio（リアルタイムスケジューラなし）が最悪
5. RMW/DDSスレッドの優先度をコールバックより高く設定しないとデッドラインを満たせないことも確認

### 4.2 自動運転ケーススタディ（ALKS）

CARLAシミュレータ上のAutomated Lane Keeping System（3つのR2Rノード：FlexRayアダプタ・ALKS制御・ビジュアライザ）で評価。全ノードでSCHED_FIFOを使用。

3ノードにまたがるオドメトリチェーンの測定結果から、チェーン全体のレイテンシがIMUコールバック内のネットワーク通信遅延に支配されること、ALKSノードの重いタイマー処理（MPC計算）がオドメトリチェーンのlatencyに影響しないこと（別スレッド・別優先度で動作するため）が確認された。提案構造が複数ノードにまたがる実用的アプリケーションでも決定論的動作を実現できることを実証した。

## 5. 議論

**R2Rの利点：** サンプリングとコールバック実行の完全分離、コールバックのスケジューラビリティ分析がROSサンプリングから独立、非同期スタイルによる複数トピックデータ統合の容易さ、ROS以外のイベントソースとの統合、epoll/io_uringの利用可能性。

**R2R固有の課題：** Rustのボローチェッカーによる共有データ構造化の制約（不適切な構造化がタイミング悪化につながる）、チャネル容量固定（11）による低優先度タスクでの溢れリスク。

## 7. 結論と今後の課題

- R2Rのデフォルト構成ではリアルタイム性を保証できない
- 提案する**futures-rt構造**（サンプリング・DDS高優先度化 + コールバックの低優先度個別スレッド実行）が全評価バリアント中で最良
- C++のrclcpp-rtと同等の性能を達成し、理論的RTA値とも一致
- 自動運転ALKS上で複数ノードにまたがる決定論的動作を実証

**今後の課題：** チャネル容量の設定可能化、サンプリング・RMW/DDSオーバーヘッドの詳細モデリング、R2R専用の応答時間分析手法の開発。
